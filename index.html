What is openGH?

It’s an open initiative created to be a meeting point for all code enthusiasts inside Grasshopper3d community. If you have been searching for a place where research, share and learn about coding related to Grasshopper with other people around the world, this is it.

How could I contribute?

The first thing you need to start a project is a personal GitHub account. Then, just send and email to openghdev@gmail.com with your project proposal and we will open a new repository (with a small description and README and LICENSE files) giving you full administration permissions.

After that you are responsible for managing the repository, granting access and permissions for every new participant.

The OpenGH team will be there to help you as far as we can in any trouble you have with the repository administration. Our idea is to make a self-managed community, the coding translation of the official Grasshopper3d forum.

How will be the projects organized? Will I have access to all projects in the GitHub organization?

Every project proposal will be translated into an online repository inside openGH GitHub organization profile administered by the applicant/s; they will grant Push, Pull and Administration permissions in the repository. Every new participant will be able to access to any permission roll (from “Only Pull” to “Administration”), this must be coordinated by the project founders.

If you want to participate in several projects in the community, just send an email to the contact address provided by the repositories administrators. You can participate in all the projects you want. And remember that the open-source profile of every repository in openGH will always let you to fork and download it.

What kind license is going to be used in the developed projects?

It will a standard Mozilla open source license. Some projects profiles (like wrappers, ports and streamline) will be linked to existing licenses that comes from the code that they derive from. Just take some minutes to read the Mozilla license and the one inherent to the existing code.

We have chosen Mozilla licensing model because it’s a nice compromise between a totally permissive license (like the MIT license) and a little bit more restrictive license (like BSD licenses). We want to protect as far as we can’t the code generated in openGH keeping its core opensource for as long as possible (even if some projects in the future derive in propietary license models), but giving you freedom to choose how your project will evolve in every moment.

We'll not control how your project and needs evolve, but we will encourage you to stay as much as you can in the open source side. If for some reason your project turn into a non-opensource license model, just remember that all the information generated until that point will remain public and with the starting license.

That's because we want to turn openGh into a big code library for learning,researching and fun.

You will be able to choose if OpenGh appears in your project credits or not, that is up to you. But you will always have to include all the people that have contributed in your project.

Why have you chosen GitHub?

  • McNeel has a repository library where it hosts several of its open source projects such as Rhinocommon, RhinoPython, and several languages of code samples.
  • The process for configuring your machine to work with Github has been very streamlined. Download an app (available for Windows, Mac and Android) and clone a repository to your local machine. More complicated issues, such as conflicts, can be dealt with rather easily. All of this is very well documented.
  • The Github support is very quick to reply in case there are any issues.
  • Github is free for open source projects with 1Gb of storage “soft limit”.
  • Dale Fugier has a repository library there! https://github.com/dalefugier
  • Gist for quick code snippets
  • GitHub Pages to create quick pages for your projects.
  • All this GH addons developers use GitHub: Daniel Picker, Nathan Miller, Ursula Frick and Thomas Grabner (a.k.a. UTO), Arthur Mamou-Mani, Giulio Piacentino, Luis Fraguada...

What kind of projects are suitable to be inside openGH?

  • Wrappers - Projects that wrap existing libraries to expose their functionality to Grasshopper.

  • Ports - Projects that port code from other languages to a GH compatible language.
  • Suite - Projects that set out to compose a new set of components, including custom datatypes for specific data interchange.
  • Libraries - .dll or other referenced assembly for classes to be implemented in other add-ons.
  • Streamline - Projects that aim to create a comprehensive set of components from existing add-ons that have significant functional redundancy.

What programming languages could be used to develop projects?

Any language supported by GH (C#, VB.NET and python area allowed). Just choose one for your project and start coding.

Compiled and VB/C#/Python custom components are allowed. Every development team must choose the coding language/s used as well as the distribution format.

In the first months we will suggest to the teams to develop code using standard VB, C# and Python Grasshopper custom components. This will let us to test the system and open the project to a wider range of users with different knowledge levels.

For non-compiled projects we suggest to upload both, the .gh file with the custom component and the exported code file into the proper format (.vb, *.py,...).
